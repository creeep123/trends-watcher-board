# 代理 IP 轮换设计文档

**日期**: 2026-03-22
**作者**: Claude + Happy
**状态**: 设计阶段

## 问题描述

Google Trends API 返回 429 错误（Too Many Requests），对单个 IP 进行限频。当前请求间隔已设置为 2 秒，但仍被限频。

## 目标

实现免费的代理 IP 轮换机制，在遇到限频时自动切换代理，确保 Google Trends 数据获取的可用性。

## 约束条件

- **纯免费方案**：不使用任何付费代理服务
- **用户体验**：可接受 10-15 秒等待时间，但需要显示实时状态
- **自动化**：无需手动维护代理列表
- **最小改动**：基于现有代码结构进行扩展

## 方案选择

**方案 A（已选）**：GitHub 代理源 + 懒加载验证

- 从 GitHub 上的免费代理列表获取代理
- 懒加载验证：使用时才验证，可用则标记
- 定期清理：每小时清理未验证的过期代理

### ProxyManager 组件

#### 接口设计

```python
class ProxyManager:
    def get_proxies(self, count: int = 3) -> list[str]
        """获取 N 个可用代理（优先返回已验证的）"""

    def mark_failed(self, proxy: str) -> None
        """标记代理失败，增加失败计数"""

    def mark_success(self, proxy: str) -> None
        """标记代理成功，重置失败计数"""

    def refresh_list(self) -> int
        """从 GitHub 刷新代理列表，返回新增数量"""

    def get_status(self) -> dict
        """获取代理池状态统计"""
```

#### 代理池结构

```python
_proxy_pool: dict[str, ProxyInfo] = {}

class ProxyInfo:
    verified: bool      # 是否已验证可用
    last_check: float   # 上次验证时间（timestamp）
    fail_count: int     # 连续失败次数
    last_success: float # 上次成功时间
```

#### 验证逻辑

- **超时时间**：5 秒
- **测试方法**：`https://www.google.com` 的 HEAD 请求
- **移除条件**：连续失败 3 次
- **过期清理**：每小时清理超过 1 小时未验证的未验证代理

## 数据流

### 正常流程（无限频）

```
用户请求 → fetch_related_queries()
    ↓
直接创建 TrendReq（无代理）
    ↓
成功返回数据（2-3秒）
```

### 限频流程（使用代理）

```
用户请求 → fetch_related_queries()
    ↓
直连请求返回 429
    ↓
ProxyManager.get_proxies(count=5)
    ↓
依次尝试每个代理：
    - 创建带代理的 TrendReq
    - 执行请求
    - 成功 → mark_success() → 返回数据
    - 失败 → mark_failed() → 尝试下一个
    ↓
全部失败 → 返回缓存数据 + _stale: true
```

### 懒加载验证流程

```
ProxyManager.get_proxies()
    ↓
检查代理池：
    - 优先返回已验证的代理（verified=True）
    - 如果已验证代理不足，返回未验证的
    ↓
TrendReq 使用代理发起请求
    ↓
成功 → ProxyManager.mark_success() → 标记 verified=True
失败 → ProxyManager.mark_failed() → 增加失败计数
```

## 错误处理

### 场景 1：GitHub 源不可用
- **处理**：使用上次缓存的代理列表
- **日志**：记录警告，不中断服务

### 场景 2：所有代理失败
- **处理**：返回缓存数据 + `_stale: true`
- **前端显示**："暂时使用缓存数据（限频中）"

### 场景 3：代理验证超时
- **处理**：标记失败，继续下一个代理
- **超时时间**：5 秒

### 场景 4：pytrends 其他错误
- **处理**：记录错误日志，继续下一个代理
- **不标记失败**：可能是临时网络问题

## 性能考量

### 资源消耗

- **网络流量**：
  - 启动时获取代理列表：~50 KB
  - 每小时自动刷新：~50 KB
  - 月流量：< 100 MB（可忽略）

- **内存占用**：
  - 代理池（150 个）：~50 KB
  - 可忽略不计

- **CPU 占用**：
  - 验证操作异步进行，不阻塞主请求
  - 可忽略不计

### 响应时间

- **最佳情况**（直连成功）：2-3 秒
- **中等情况**（第一个代理成功）：5-10 秒
- **最坏情况**（尝试多个代理）：10-15 秒
- **降级情况**（返回缓存）：< 1 秒

## 风险和缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 免费代理质量差 | 成功率低 | 高 | 使用多个源，维护较大代理池 |
| GitHub 源失效 | 无法更新代理 | 中 | 缓存上次列表，使用多个备用源 |
| 代理验证慢 | 延迟增加 | 中 | 懒加载，只验证使用的代理 |
| 所有代理都失败 | 无法获取数据 | 低 | 降级到缓存数据 |

## 后续优化方向

1. **持久化缓存**：将验证通过的代理保存到文件，重启后快速加载
2. **地理位置优化**：优先使用特定国家的代理
3. **智能选择**：根据历史成功率排序代理
4. **自适应超时**：根据代理响应时间动态调整超时
